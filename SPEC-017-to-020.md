## Goal
System Administrator (IT/BIDB) registers new staff accounts, assigns RBAC roles, and deactivates leavers. All actions audit-logged. Temporary credentials sent via email (SRS SR3). Email must be @iyte.edu.tr domain (SRS EX-02).

# SPEC-018 — Configure Application Dates
## Maps to: UC-07-02

## Goal
System Administrator sets the application window start + deadline. System uses server-side time (NOT client time) to open/close the portal (SRS SR1). Deadline enforcement: no submissions accepted after deadline, even in-progress uploads (SRS EX-02). Manual extension flow supported (SRS AC-01).

## Depends On
SPEC-017

---

## Tasks

### Task 1 — PeriodService
File: `backend/app/services/period_service.py`

```python
class PeriodService:
    async def create_period(
        self, label: str, opens_at: datetime, closes_at: datetime,
        created_by: UUID
    ) -> ApplicationPeriod
    # Validate: closes_at > opens_at → 422 "End date cannot be before start date" (SRS EX-01)
    # Validate: no overlapping active period
    # AuditLog action="PERIOD_CREATED"
    # Saved ≤ 3 seconds (SRS perf)

    async def is_open(self, period_id: UUID) -> bool
    # Uses server-side NOW() — NEVER client time (SRS SR1, critical)

    async def extend_deadline(
        self, period_id: UUID, new_closes_at: datetime, by: UUID
    ) -> ApplicationPeriod
    # Handles SRS AC-01 (deadline extension while period is active)
    # Immediately updates portal countdown
    # AuditLog action="PERIOD_EXTENDED"

    async def emergency_close(
        self, period_id: UUID, by: UUID
    ) -> ApplicationPeriod
    # Handles SRS AC-02
    # Closes immediately regardless of time
    # AuditLog action="PERIOD_EMERGENCY_CLOSED"

    async def activate_period(self, period_id: UUID, by: UUID) -> ApplicationPeriod
    async def deactivate_period(self, period_id: UUID, by: UUID) -> ApplicationPeriod
```

Deadline enforcement at submission time: `ApplicationService.submit_application()` calls `PeriodService.is_open()` using server time. In-progress upload when deadline hits → upload finishes but Submit button blocked (SRS EX-02).

# SPEC-020 — Notification Worker: Email Delivery & Retry
## Used by: ALL specs that trigger notifications

## Goal
All email notifications go through Celery workers. Failed deliveries retry with exponential backoff (60s, 120s, 240s, 480s, 960s). `task_acks_late=True` prevents task loss on worker crash. 1000 notifications dispatched within 5 minutes (SRS UC-03-02 SR1).

## Depends On
SPEC-001

---

## Tasks

### Task 1 — NotificationService
File: `backend/app/services/notification_service.py`

### Task 2 — Celery Task
File: `backend/app/workers/notification_tasks.py`

```python
@celery.task(bind=True, max_retries=5, default_retry_delay=60)
def send_notification(self, notification_id: str):
    # 1. Load Notification from DB
    # 2. Render Jinja2 template (see Task 3)
    # 3. Send via SMTP (env: SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS)
    # 4. Success: status=SENT, sent_at=NOW()
    # 5. Failure: retry with exponential backoff:
    #    delay = 60 * (2 ** self.request.retries)  → 60, 120, 240, 480, 960s
    # 6. After max_retries: status=FAILED, log error
```
